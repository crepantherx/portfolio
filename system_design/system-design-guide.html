How to Approach Any Design
1. Clarify Requirements (5 min)
Functional vs Non-functional. DAU, QPS, storage needs. SLA, consistency model. Don't guess — ask explicitly. State every assumption.
2. Capacity Estimation (5 min)
Write numbers out loud. Storage/day, read:write ratio, bandwidth. Back-of-envelope before architecture — it drives decisions.
3. High-Level Design (10 min)
Core data flow. APIs, DB schema sketch. Critical path only. No over-engineering yet.
4. Deep Dive: Bottlenecks (15 min)
Pick 2–3 hard parts. Fan-out, sharding key, cache invalidation, consistency model. Show tradeoff awareness.
5. Operational Concerns (5 min)
Failure modes, monitoring, rate limiting, gradual rollout. This separates senior candidates.

Key Design Decisions

Why GPU workers pull from a queue instead of direct HTTP? ▾
GPU inference is expensive and slow (~seconds). Accepting all requests directly would exhaust connection pools. A queue decouples acceptance from processing, lets you prioritize paying tiers, and lets workers pull at their own pace — critical for backpressure. Workers process one request at a time per GPU slot; any idle capacity is immediately consumed from queue.
How does prompt caching work? ▾
LLM inference has a prefill phase (processing the full input) and a decode phase (generating tokens). The prefill KV cache (key-value attention matrices) for a prompt can be stored and reused. If the same system prompt prefix is reused (common for API customers with fixed system prompts), the prefill computation is skipped — reducing latency by 40–60% and compute cost proportionally. Cache is stored in GPU VRAM or fast NVMe, keyed by a hash of the prefix.
Multi-tenancy: how are users isolated? ▾
At the API gateway level: each API key maps to an organization. Rate limits are enforced per key (token bucket, 2-tier: per-minute and per-day). Queues are logically separated per tier (free, plus, enterprise) with priority scheduling. Cost is tracked per request in a usage DB (Postgres). GPU workers are shared but requests are tagged with org_id for billing and observability.

Why Streaming (SSE) is Critical for LLMs

An LLM generates tokens one at a time. Average latency to first token (TTFT) might be 500ms; full response for 500 tokens might take 10s. Without streaming, users see nothing for 10s then full dump — terrible UX. With SSE (Server-Sent Events), the server pushes each token as generated.

// Server-Sent Events — lightweight, unidirectional, HTTP/1.1 GET /v1/chat/completions Accept: text/event-stream Authorization: Bearer sk-... // Server responds: data: {"id":"chatcmpl-xyz","choices":[{"delta":{"content":"Hello"}}]} data: {"id":"chatcmpl-xyz","choices":[{"delta":{"content":" world"}}]} data: [DONE]
// WebSocket vs SSE SSE is unidirectional (server → client) over HTTP — simpler, load-balancer friendly, auto-reconnects. WebSocket is bidirectional but adds complexity. For token streaming, SSE wins. Use WebSocket for ChatGPT-style interactive features like voice or tool calls.

Streaming Architecture Challenge: Sticky Sessions

The GPU worker generating tokens must hold the connection open to stream back to the same client. This means connection affinity: the API gateway must route the response back to the original connection. Solutions: (1) the worker pushes tokens to a Redis pub/sub channel keyed by request_id, and a stateless streamer server subscribes and forwards to the SSE connection; (2) direct long-polling with the worker node. Option 1 is preferred at scale.

Large models (GPT-4 class: ~1T params) don't fit on a single GPU. Two strategies:

Feed Ranking Pipeline

Get ~1000 recent posts from all accounts you follow. Pulled from user post index sharded by user_id. Also includes "interest graph" candidates from accounts you don't follow (Explore/Reels).
2. Multi-stage Ranking

3. Diversity & Business Logic
De-duplicate (same author, similar content). Insert sponsored posts at fixed intervals. Filter content policy violations. Final top-N.
4. Serve from Cache
Top 50 posts cached per user. Next page lazily loaded on scroll. Cache TTL: 5–15 mins. Stale feed is acceptable (eventual consistency).

The Fan-out Problem — Celebrity Edition

// The Problem Selena Gomez posts a photo. She has 400M followers. If each post triggers 400M cache writes, that's 400M operations in seconds. At 10µs per write = 4,000 seconds of compute. Traditional fan-out-on-write breaks at celebrity scale.

Two Fan-out Strategies

Fan-out on Write (Push)

  • Write to all follower caches on post creation
  • Feed reads are O(1) — instant cache hit
  • Good for users with few followers (<10K)

Fan-out on Read (Pull)

  • Compute feed at read time by fetching posts from all followees
  • Expensive reads for users following 1000+ accounts
  • But correct for celebrities — no 400M writes

Instagram's Hybrid Approach

Regular users (<10K followers): Fan-out on write. Post lands in Kafka → fan-out workers write to Redis feed caches of all followers within seconds.

Celebrities (>10K followers): Fan-out on read. Celebrity posts are NOT pushed to follower caches. When you open your feed, the system merges (a) your pre-computed cache from regular followees + (b) live-fetched recent posts from your celebrity followees.

// Key Insight The threshold is configurable. A "hot user" flag is set when follower count exceeds threshold. Feed service knows to merge strategies. This is classic write-amplification vs read-amplification tradeoff.

Key Problems

Who viewed my story — at 100M viewers? ▾
Problem: Storing every viewer for a celebrity story = 100M rows per story. Solution: For normal users, store exact viewer list in Redis set. For celebrities, use a HyperLogLog (probabilistic cardinality estimator, ~12KB, ~0.81% error) for count, and a Cassandra table for recent viewers only. The exact viewer list is truncated after 1000 entries for celebrities — Instagram doesn't show you if Kylie Jenner saw your story, and vice versa they cap the displayed viewer list.
How are stories expired efficiently? ▾
Two-pronged: (1) TTL in Redis/Cassandra: set expiry on insertion. Storage auto-reclaims. (2) Soft-delete + background job: a scheduled job runs every hour, deletes media from object store, removes from all follower story queues. On read, stale story IDs are checked and skipped. Object store deletion is delayed 48h for replay/appeals.

Why WebSocket (not HTTP polling)?

HTTP polling (client checks every N seconds) wastes bandwidth and adds latency. WebSocket maintains a persistent bidirectional TCP connection — server can push messages to client at any time. Each server handles ~100K concurrent WebSocket connections. With 2B users and ~20% online simultaneously = 400M connections = ~4000 servers minimum (just for connections).

// Why Erlang WhatsApp was built on Erlang/BEAM. Erlang's lightweight processes (~500 bytes each), non-blocking I/O, and supervisor trees make it ideal for millions of concurrent connections. A single 20-core box could handle 2M connections. This is why 50 engineers ran 2B-user scale before Facebook acquisition.

Exactly-Once Delivery Semantics

1. Sender → Server (sent ✓)
Phone A sends message. Server acknowledges with message_id. App shows single grey tick. Message stored in Cassandra with status=PENDING.
2. Server → Recipient (delivered ✓✓)
Server pushes to recipient's WebSocket. Recipient client ACKs. Server updates status=DELIVERED. Notifies sender → double grey ticks.
3. Recipient reads (read ✓✓ blue)
User opens chat → client sends READ receipt. Server updates → blue ticks shown to sender. Read receipts can be disabled — then server never sends that event.

Offline Message Handling

If Phone B is offline: message stored in offline queue (Cassandra, TTL = 30 days). When Phone B comes online, on WebSocket connect, server pushes all queued messages in order. Client ACKs each; server marks delivered and flushes queue.

// Message Ordering Guarantee Messages use a per-conversation sequence number (not timestamp) to handle clock skew across devices. Cassandra stores messages by (conversation_id, seq_num) — guarantees order even with out-of-order network delivery.

Media Messages

Images/videos are NOT stored on WhatsApp servers long-term. Flow: (1) Phone A encrypts media client-side → uploads to WhatsApp CDN (BLOB server). (2) Message contains encrypted URL + decryption key. (3) Phone B downloads and decrypts. Media is deleted from CDN after delivery or after 30 days. This keeps server storage manageable — WhatsApp doesn't store your media.

Group Delivery Receipt Problem

For individual messages: 1 delivered receipt. For 1024-member group: 1024 receipts. Aggregating them: server must track per-(message_id, member_id) delivery. At 100M group messages/day × 500 avg members = 50B delivery events/day. Solution: aggregate in Redis first (counter per message), persist to Cassandra asynchronously in batches.

// Delivery vs Read at Group Scale WhatsApp shows "delivered to N, read by M" not individual read status in groups. Counting is done with approximate counting (counter increments) not full member enumeration. Exact per-member read receipts shown only in Info page, fetched lazily on demand.

End-to-End Encryption (Signal Protocol)

WhatsApp uses the Signal Protocol. Server never sees plaintext messages.

Key Registration
On install, device generates identity key pair, signed prekey, and 100 one-time prekeys. Public keys uploaded to WhatsApp key server. Private keys NEVER leave device.
Session Establishment (X3DH)
First message: sender fetches recipient's prekeys from server. Performs X3DH (Extended Triple Diffie-Hellman) key agreement → shared secret. Derives message encryption key. Consumes one-time prekey (deleted from server).
Double Ratchet Algorithm
Every message derives a new key using Double Ratchet. Forward secrecy: compromise of current key can't decrypt past messages. Break-in recovery: compromise can't decrypt future messages after ratchet advances.
Group Keys (Sender Keys)
For groups: each sender distributes a "Sender Key" to all members. Only one encryption per message (not N). New member = new Sender Key distribution (forward secrecy maintained).
// What WhatsApp CAN See Metadata: who messaged whom, when, how often, group membership, IP addresses, phone numbers. Not message content or media. This metadata alone is extremely valuable for intelligence/ad purposes.

Driver Matching Algorithm

1. Find Candidates
Query Redis geo index for all available drivers within N km radius of rider. Start with 1km, expand to 3km, 5km if not enough candidates. Return up to 50 candidates.
2. Compute ETA (not straight-line distance)
Critical: straight-line distance is useless in a city. Real road-network ETA computed for top 10 candidates using pre-computed routing graph. Uber H3 (hexagonal grid) hierarchically narrows routing candidates.
3. Rank Drivers
Score = f(ETA, driver rating, acceptance rate, trip duration estimate). ML model trained on historical completions. ETA is weighted 80%.
4. Offer & Timeout
Best driver gets trip offer (15-second window). If declined or timeout, offer to next candidate. Concurrently offer to top-3 to reduce latency — first accept wins, others notified.
// Consistency Challenge Two riders could both match the same driver simultaneously. Uber uses optimistic locking on driver status: driver row has version column. UPDATE driver SET status=DISPATCHED WHERE id=X AND status=AVAILABLE AND version=N — if 0 rows updated, driver taken, retry with next candidate.

Geospatial Indexing — The Core Problem

Finding nearest drivers efficiently. Naive solution: scan all 5M drivers, compute distance — O(N), too slow.

Geohash

Encode lat/lng into a string where common prefix = geographic proximity. Precision levels:

Geohash LengthAreaUse
4 chars~40km × 40kmCountry-level
6 chars~1.2km × 0.6kmNeighborhood
8 chars~38m × 19mDriver matching

Drivers indexed in Redis sorted set by geohash. To find nearby drivers: compute rider's geohash + 8 neighboring cells (to avoid edge-case where nearby driver is in adjacent cell). Redis GEORADIUS command does this natively.

H3 — Uber's Hexagonal Grid

// Why Hexagons over Geohash Hexagons have equal-distance neighbors (all 6 neighbors equidistant from center). Squares/rectangles have 8 neighbors with 2 different distances (orthogonal vs diagonal). For spatial operations like surge pricing zone calculation and demand heatmaps, hexagons are more accurate. Uber open-sourced H3.

H3 hierarchical resolution: resolution 9 cells (~0.1 km²) for matching, resolution 7 (~5 km²) for surge zones, resolution 4 (~288 km²) for city-level analytics.

Location Update at 20M writes/sec

Each of 5M drivers sends GPS every 250ms = 20M writes/sec. Strategy: (1) Driver app sends to location kafka topic. Kafka absorbs the burst. (2) Location consumer updates Redis geo index and writes to Cassandra for trip replay/audit. (3) Redis TTL: driver location expires after 60s of no update (driver went offline). Never write to primary DB directly from driver app — Kafka acts as buffer.

Idempotency — The Single Most Critical Concept

// The Failure Scenario Client sends payment request. Network times out at 28s. Client retries. Server already processed the first request (it just took long). Without idempotency, customer is charged twice.

Solution: Idempotency Keys

POST /v1/payments Idempotency-Key: "client-generated-uuid-abc123" { "amount": 5000, "currency": "USD", "source": "card_token_xyz" } // Server logic: processPayment(idempotency_key, request): existing = redis.get("idem:" + idempotency_key) if existing: return existing.response // replay cached response // Atomic lock — prevents concurrent duplicate execution lock = redis.set("idem:" + idempotency_key, "PROCESSING", NX, EX=300) if !lock: return "409 Conflict - concurrent request" result = executePayment(request) redis.set("idem:" + idempotency_key, result, EX=86400) return result
// TTL on Idempotency Keys Keys expire after 24h. After that, the same key could trigger a new charge — clients must use fresh keys for new payments. Keys should be client-generated UUIDs (not server), ensuring uniqueness without coordination.

At-least-once vs Exactly-once

Kafka delivers at-least-once. Payment events may be delivered twice. At the consumer (ledger update), use the same idempotency_key to prevent double-crediting: INSERT INTO ledger ... ON CONFLICT (idempotency_key) DO NOTHING.

Distributed Transactions — The Transfer Problem

Debit account A, credit account B. If they're in different databases/services, you can't use a single ACID transaction.

Two-Phase Commit (2PC) — What Banks Used To Do

2PC Pros

  • True atomicity — all or nothing
  • Strong consistency guarantee

2PC Cons

  • Blocking: if coordinator crashes, participants block forever
  • Low throughput — tight lock coordination
  • Not practical across microservices at scale

SAGA Pattern — Modern Solution

1. Debit sender
Execute local transaction, publish "SenderDebited" event to Kafka.
2. Credit receiver (on event)
Payment service consumes event, credits receiver account, publishes "ReceiverCredited".
3. Failure → Compensating Transaction
If step 2 fails, publish "CreditFailed" → compensating transaction reverses step 1 (re-credits sender). Compensation must also be idempotent.
// SAGA Tradeoff No true atomicity — briefly inconsistent state is visible (sender debited but receiver not yet credited). Acceptable for payments if eventual consistency window is short (<100ms). Not acceptable for inventory (oversell). Know when to use which.

Why Naive Single-PUT Upload Fails

// Problems with single HTTP upload (1) Network interruption at 99% → restart from 0. (2) 10GB through your backend server = bottleneck, memory pressure. (3) No progress feedback. (4) Load balancer / proxy timeouts (default 60s). (5) No parallelism. For 10GB at 10 Mbps = 133 minutes — any dropout restarts.

Requirements for Production Upload

  • Resumable: Network drops mid-upload → resume from last successful chunk
  • Parallel: Upload multiple chunks simultaneously for speed
  • Bypass backend: Direct upload to object store — don't saturate your servers
  • Progress: Client shows real % progress
  • Deduplication: Same file uploaded twice → don't store twice (content hash)

Chunked Multipart Upload Protocol

// Phase 1: Initiate upload POST /api/upload/initiate { file_name, file_size, content_type, chunk_size: 5242880 } // 5MB chunks // Server response { upload_id: "upl_abc123", total_chunks: 2048 } // Phase 2: Upload chunks (parallel) for chunk_index in [0..2048] // 16 concurrent uploads: PUT /api/upload/{upload_id}/chunk/{chunk_index} Content-Range: bytes 0-5242879/10737418240 Body: [binary chunk data] // Server stores chunk, returns checksum { etag: "md5-abc...", chunk_index: 0 } // Phase 3: Complete POST /api/upload/{upload_id}/complete { parts: [{ chunk_index: 0, etag: "..." }, ...] } // Phase 4: Resuming after failure GET /api/upload/{upload_id}/status { completed_chunks: [0,1,2,5,6], missing: [3,4,7..2047] } // Client re-uploads only missing chunks

Chunk Storage Tracking

-- Upload state table uploads: (upload_id, user_id, filename, total_chunks, status, created_at) upload_chunks: (upload_id, chunk_index, etag, size, uploaded_at) -- Check completeness SELECT COUNT(*) FROM upload_chunks WHERE upload_id = ? -- == total_chunks → trigger assembly job
// Chunk Size Tradeoff Smaller chunks = more overhead (more HTTP requests, more DB rows). Larger chunks = more to re-upload on failure. Industry standard: 5–10 MB per chunk. AWS S3 minimum part size is 5MB.

Traffic Surge Handling

Black Friday, Super Bowl, product launches — 100x normal traffic in seconds. How to survive without burning money.

Tiered Defense Strategy

Tier 1: CDN Absorbs Static Load
80% of traffic is static assets (JS, CSS, images, HTML shells). All served from CDN edge (Cloudflare/Fastly). These requests never hit your origin. CDN handles essentially unlimited scale. Your task: cache everything possible.
Tier 2: Rate Limiting & Traffic Shaping
API gateway enforces per-IP, per-user rate limits. Token bucket algorithm. Burst allowance for legitimate spikes, sustained rate limit for abuse. Queue excess requests don't reject them — return 429 with Retry-After header.
Tier 3: Horizontal Auto-scaling
Kubernetes HPA scales stateless services on CPU/RPS. Pre-warm: for known events (Super Bowl kickoff), scale out 15 mins early. Cold start is too slow — pre-provisioned instances. AWS Target Tracking: maintain 60% CPU, scale when breached.
Tier 4: Circuit Breakers & Graceful Degradation
If DB is overwhelmed: serve stale cached data. If recommendation service is down: show popular items. If payment service is slow: queue the request, don't timeout user. Always define what "degraded" looks like before it happens.
Tier 5: Load Shedding (Last Resort)
If all else fails, reject non-essential requests explicitly. Show "We're experiencing high demand" page (served from CDN, no backend needed). Priority: paying customers > free users > unauthenticated. Queue for virtual waiting room.

Queue-Based Virtual Waiting Room

Used by Ticketmaster, government vaccine booking sites. Instead of 503 errors:

What "Realtime Feed" Actually Means

Instagram doesn't show you posts the millisecond they're posted. The feed is refreshed when you (a) pull to refresh, (b) tap "New Posts" banner, or (c) app foregrounds. The "realtime" element is the notification that new posts exist, not the feed itself being pushed.

The banner "2 New Posts" = a push notification (APNs/FCM) or SSE event sent to all online followers. Actual feed refresh = client re-fetches from API (with updated pre-computed cache).

Typeahead Autocomplete (<50ms)

Requirements: sub-50ms, personalized, handles typos, trending boosts.

Trie-Based Approach (in-memory)

// Trie node with frequency count TrieNode { char, children: {}, frequency, is_end } search("app") → traverse: a → p → p (prefix node) DFS from prefix node → ["apple"(1M), "app"(500K), "application"(200K)] return top-5 by frequency // Updated on each search to boost trending terms

Problem: full trie in-memory for all search terms = 10s of GB. Not practical at Google scale. Solution: use Redis sorted sets for popular prefix caching.

Redis Prefix Cache Approach

// For each term, store all prefixes in sorted set scored by frequency ZADD autocomplete:a 1000000 "apple" ZADD autocomplete:ap 1000000 "apple" ZADD autocomplete:app 1000000 "apple" ZADD autocomplete:appl 1000000 "apple" // Query: ZREVRANGE autocomplete:app 0 4 // O(log N) — sub-millisecond // → ["apple", "application", "apple music", ...]
// Tradeoff Redis approach uses O(L × N) storage (L = avg term length, N = terms). For 100M terms, avg length 8: 800M Redis entries. At ~50 bytes each = 40GB Redis — feasible on a cluster. Trie is more memory efficient but harder to distribute.

Token Bucket — Best for APIs

Bucket holds N tokens. Each request consumes 1 token. Tokens refill at rate R/second. Allows burst up to N, then limits to sustained R.

class TokenBucket: tokens: float = capacity last_refill: float = now() consume(n=1) → bool: elapsed = now() - self.last_refill self.tokens = min(capacity, self.tokens + elapsed * refill_rate) self.last_refill = now() if self.tokens >= n: self.tokens -= n return True # allowed return False # rate limited // Example: capacity=100, refill=10/sec // Can burst 100 requests, then sustained 10/sec
// Why Token Bucket It allows natural bursts (user activity is bursty). Smooth rate limiting that doesn't penalize legitimate burst patterns. Used by AWS API Gateway, Stripe.

Sliding Window Log — Most Accurate

// Store timestamps of all requests in last window is_allowed(user_id, limit=100, window=60): now = time.time() key = f"rl:{user_id}" // Remove expired entries redis.zremrangebyscore(key, 0, now - window) count = redis.zcard(key) if count >= limit: return False redis.zadd(key, {now: now}) redis.expire(key, window) return True // Memory: O(limit) per user (at most `limit` entries)

Most accurate but uses more memory. For 100K users with limit=1000: 100M Redis entries = ~10GB. Sliding window counter (approximate) uses less memory.

Fixed Window Counter — Simple, Has Edge Case

is_allowed(user_id): key = f"rl:{user_id}:{current_minute}" count = redis.incr(key) redis.expire(key, 60) return count <= 100
// The Boundary Bug Limit: 100 req/min. User sends 100 at 00:59 and 100 at 01:01. Both are within their respective minute windows (allowed) but 200 requests hit your API in 2 seconds. Fixed window allows 2× burst at boundaries.

Sliding window log or sliding window counter fix this. The counter approach uses Redis INCR with a rolling key — 2x the storage but much less than log.

Distributed Rate Limiting

Multiple API gateway nodes. User's requests hit different nodes. Local in-memory counters are wrong — each node sees only its fraction of traffic.

Solutions

Centralized Redis (most common) ▾
All nodes query central Redis. Atomic INCR + EXPIRE. Accurate. Adds ~1ms per request (Redis round-trip). Risk: Redis becomes single point. Mitigate with Redis Cluster + local fallback. Used by Nginx rate limit module, Kong.
Local approximation with sync ▾
Each node maintains local counter. Periodically (every 100ms) sync to central store and fetch updated global count. Allows brief over-limit (within 100ms window) — acceptable for most use cases. Huge latency win: no Redis on hot path.
Sticky sessions (simple but limited) ▾
Route each user to the same gateway node via consistent hashing on user_id. Local counters are accurate. Problem: uneven load, hot users, node failure loses counter state. Only viable at small scale.

DB Sharding & Replication

When a single database can't hold the data or handle the QPS. Horizontal partitioning done right.

When to Shard

// Don't shard prematurely Every additional shard adds: cross-shard query complexity, rebalancing overhead, distributed transaction pain. Single Postgres with read replicas handles most workloads to 100M users. Shard only when you can quantify the bottleneck.

Indicators: (1) Single DB CPU/IO consistently >70%. (2) Table size >1TB. (3) Write throughput saturated even with connection pooling. (4) Hot row contention (everyone writes to same user row).

Message Queues & Kafka

Async decoupling, guaranteed delivery, ordering, consumer groups, and why Kafka beats Redis for durability.

Why Message Queues

Synchronous (Direct HTTP)

  • Simple, immediate response
  • Easy to trace/debug
  • Tight coupling: if downstream is slow, upstream waits

Async (Queue)

  • Decoupled: producer doesn't wait
  • Buffer against traffic spikes
  • Retry on failure without client involvement
  • Fan-out to multiple consumers

Key Properties

  • Partitions = parallelism: More partitions = more consumers = higher throughput. Each partition is consumed by exactly one consumer in a group.
  • Offset tracking: Consumers track their own position (offset) per partition. On failure, restart from last committed offset. No data loss.
  • Durability: Messages written to disk, replicated across brokers. Retained for configurable period (default 7 days). Can replay history.
  • Ordering: Strict ordering within a partition. No global ordering across partitions. Choose partition key carefully (e.g., user_id → same user's events always ordered).

Distributed ID Generation

Globally unique IDs without coordination. Snowflake IDs, ULIDs, and why auto-increment breaks at scale.

Why Auto-increment Fails

// Auto-increment Problems (1) Single DB sequence = bottleneck (only primary DB can generate IDs). (2) Reveals business metrics (order #1000 tells competitors scale). (3) Insecure/guessable (user_id=1, 2, 3...). (4) Cross-shard conflicts when merging DBs.

Twitter Snowflake — The Industry Standard

64-bit Snowflake ID Layout
41 bits
timestamp
10 bits
machine ID
12 bits
sequence
= 41 + 10 + 12 = 63 bits usable (1 sign bit = 0)
// Snowflake ID properties: timestamp_bits = 41 // 69 years from epoch (custom epoch: ~2010) machine_bits = 10 // 1024 unique machines/datacenters sequence_bits = 12 // 4096 IDs per millisecond per machine // Throughput: 1024 machines × 4096/ms = 4.1M IDs/ms = 4.1B IDs/sec def generate_id(): ts = current_ms() - EPOCH seq = atomic_increment(sequence) % 4096 if same_ms_as_last: seq = next # increment within same ms else: seq = 0 # reset on new ms return (ts << 22) | (machine_id << 12) | seq // IDs are k-sortable by time (important for DB indexing!)

Why Sortability Matters

Random UUIDs (v4) cause index fragmentation — each insert is at a random B-tree position, causing page splits and cache misses. Snowflake IDs are monotonically increasing (k-sorted) — new inserts always go to the end of the B-tree. 10–100× faster insert performance on large tables.

ULID — Snowflake Alternative

// ULID: 128-bit (26 Crockford base32 chars) 01ARZ3NDEKTSV4RRFFQ69G5FAV // First 10 chars = timestamp (ms precision) // Last 16 chars = random // URL-safe, case-insensitive, sortable // No machine coordination needed (random component)

ULIDs are simpler to implement (no machine registration), but slightly more collision-prone (random vs sequence). For most applications, ULIDs are easier to operate. Snowflake is better for very high single-machine throughput.
