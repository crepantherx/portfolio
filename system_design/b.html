ChatGPT: Comprehensive System Design

A deep-dive into the architecture, infrastructure, and design decisions behind a production-grade large language model (LLM) chat service serving 100M+ users. Every component is explained with the rationale behind each choice.

1. High-Level Architecture Overview

ChatGPT is a multi-tier, globally distributed, real-time conversational AI platform. It serves millions of concurrent users with sub-second latency for token-by-token streaming responses. The system spans:

  • Client Layer – Web (React), iOS/Android (Swift/Kotlin), API
  • Edge / CDN – Cloudflare for static assets, DDoS protection, regional routing
  • API Gateway – Envoy-based, handles auth, rate-limiting, request routing
  • Backend Services – Orchestration, session management, conversation state
  • Inference Cluster – GPU pods running optimized LLM inference (vLLM / TensorRT-LLM)
  • Data Plane – PostgreSQL, Redis, Kafka, S3-compatible object storage
  • ML Platform – Training (pre-training, fine-tuning, RLHF), evaluation, deployment
  • Observability – Prometheus, Grafana, OpenTelemetry, centralized logging (Loki)

2. Frontend Architecture

The ChatGPT web client is a single-page application (SPA) built with React and TypeScript, emphasizing real-time streaming and optimistic UI updates.

2.1 Technology Stack & Rationale

Technology Purpose Why Chosen
React 18+ UI framework Concurrent rendering for smooth streaming; vast ecosystem; Server Components possible for SSR
TypeScript Type safety Catch errors at compile time; self-documenting API contracts; better IDE support
Zustand State management Lightweight (no boilerplate like Redux); excellent for managing chat threads and streaming tokens
Server-Sent Events (SSE) Streaming protocol Unidirectional server→client; native browser support; simpler than WebSockets for token streaming
WebSocket (fallback) Bidirectional real-time Used for features like shared conversations, typing indicators, or when SSE is blocked by proxies
IndexedDB Local persistence Offline conversation drafts; caching conversation history locally for instant load
🔑 Key Decision

Why SSE over WebSockets for Token Streaming?

Token-by-token generation is inherently unidirectional (server → client). SSE is simpler, uses standard HTTP/2 or HTTP/3, works seamlessly with CDNs and load balancers, and automatically reconnects. WebSockets add bidirectional overhead and stateful connection complexity that isn't needed for the primary chat flow. However, a thin WebSocket layer is maintained for features requiring client→server push (e.g., "stop generating" signals, voice mode real-time audio).

2.2 Streaming Rendering Architecture

As tokens arrive via SSE, the React component performs incremental state updates using React.startTransition to keep the UI responsive. The markdown parser (remark / rehype) runs incrementally on the accumulating text, and syntax highlighting is applied lazily via Shiki or Prism.js in a Web Worker to avoid blocking the main thread. 

            // Simplified streaming hook
            const useChatStream = (conversationId) => {
            const [tokens, setTokens] = useState<string[]>([]);
            const [isStreaming, setIsStreaming] = useState(true);

            useEffect(() => {
            const eventSource = new EventSource(
            `/api/conversations/${conversationId}/stream`
            );
            eventSource.onmessage = (event) => {
            const token = JSON.parse(event.data).token;
            startTransition(() => setTokens(prev => [...prev, token]));
            };
            eventSource.addEventListener('done', () => setIsStreaming(false));
            return () => eventSource.close();
            }, [conversationId]);
            };

3. Backend Services

The backend follows a microservices architecture with clear domain boundaries, communicating primarily via gRPC (internal) and REST/SSE (external). Each service is containerized and runs on Kubernetes.

3.1 Core Services

Service Responsibility Tech Stack
API Gateway Auth, rate limiting, routing, TLS, request/response transformation Envoy Proxy, custom Go filters
Auth Service OAuth2/OIDC, session tokens, MFA, API key management Go, PostgreSQL, Redis
Chat Orchestrator Conversation state machine, context window assembly, tool calling Go (high concurrency), gRPC
Moderation Service Content safety, toxicity classification, PII detection Python, custom fine-tuned classifiers
Inference Gateway Load-aware routing to GPU pods, batching, priority queuing Rust (for performance), gRPC streaming
User Profile Service Preferences, subscription tier, usage tracking Go, PostgreSQL
🔑 Key Decision

Why gRPC for Internal Service Communication?

gRPC offers strongly-typed contracts via Protobuf, HTTP/2 multiplexing, native bidirectional streaming (critical for token relay), and excellent performance with binary serialization. Compared to REST+JSON, gRPC reduces latency by ~30-40% for internal calls and eliminates manual serialization code. The streaming support is especially important—the Chat Orchestrator can relay inference tokens as a gRPC stream directly to the API Gateway, which then converts to SSE for the client.

3.2 Conversation State Machine

Each conversation is modeled as a finite state machine with states: IDLE → PROCESSING → STREAMING → COMPLETED | ERROR | CANCELLED. This allows the system to handle interruptions gracefully (user clicks "stop", network disconnect, moderation flag) and resume or rollback as needed. 

            // Conversation FSM states (Protobuf)
            enum ConversationState {
            IDLE = 0;        // Waiting for user input
            QUEUED = 1;      // Request queued for inference
            PROCESSING = 2;  // Model is generating
            STREAMING = 3;   // Tokens being sent to client
            COMPLETED = 4;   // Generation finished successfully
            CANCELLED = 5;   // User interrupted
            ERROR = 6;       // Failure occurred
            FLAGGED = 7;     // Moderation intervened
            }

4. Model Inference Infrastructure

ML Ops Infrastructure

This is the most computationally expensive part of the system. The inference cluster must deliver high throughput while maintaining low latency (time-to-first-token < 200ms, inter-token latency < 30ms).

4.1 GPU Cluster Design

Component Specification Rationale
GPU Nodes NVIDIA H100 (80GB) × 8 per node, NVLink interconnected H100 offers ~3x inference throughput vs A100 for transformer models; NVLink enables tensor parallelism across GPUs
Serving Engine vLLM with PagedAttention PagedAttention manages KV-cache memory in non-contiguous blocks, achieving ~24x higher throughput than naive implementations by eliminating memory fragmentation
Quantization INT8/FP8 weight-only quantization (AWQ / GPTQ) Reduces model size by ~2x with negligible quality loss; allows serving larger models or more concurrent requests per GPU
Orchestration Kubernetes with GPU operator + KubeRay Ray enables dynamic scaling of inference workers; K8s GPU operator handles driver installation and device plugin
Continuous Batching Built into vLLM Dynamically adds/removes requests from a running batch; maximizes GPU utilization without waiting for batch completion
🔑 Key Decision

Why vLLM with PagedAttention over vanilla HuggingFace Transformers?

Traditional inference frameworks allocate a contiguous KV-cache per sequence up to max_context_length. For a model with 32K context, this wastes enormous GPU memory. PagedAttention borrows virtual memory paging concepts from operating systems—it stores KV-cache in non-contiguous blocks and only allocates what's actually used. This results in 2-4× higher throughput and significantly higher GPU memory utilization. In production benchmarks, vLLM serves 24× more requests/sec than HF with the same hardware, which directly translates to lower cost per token.

5. Training Infrastructure

ML Training

Training GPT-class models involves three distinct phases, each with different infrastructure requirements:

5.1 Pre-Training (Self-Supervised)

Aspect Design Choice Explanation
Cluster Thousands of H100 GPUs across multiple nodes, InfiniBand interconnect (800 Gbps) LLM training requires massive parallelization; InfiniBand minimizes gradient synchronization latency
Parallelism 3D Parallelism: Data + Tensor + Pipeline parallelism (via Megatron-LM or DeepSpeed) Combines all three strategies to scale beyond single-node memory limits and maximize throughput
Framework PyTorch + FSDP (Fully Sharded Data Parallel) + FlashAttention-2 FSDP shards model across GPUs; FlashAttention-2 reduces memory and speeds up attention by 2-3×
Checkpointing Async distributed checkpointing to S3-compatible object storage Allows fast recovery from failures; training runs span weeks/months; fault tolerance is critical
Scheduling SLURM or Kubernetes with Volcano scheduler Gang scheduling ensures all GPUs for a job start simultaneously; preemption policies for priority

5.2 Fine-Tuning & RLHF Pipeline

🔑 Key Decision

Why RLHF (Reinforcement Learning from Human Feedback)?

Raw pre-trained LLMs are powerful but uncalibrated for helpfulness and safety. RLHF aligns the model with human values: a reward model trained on human preference comparisons predicts how "good" a response is, and Proximal Policy Optimization (PPO) fine-tunes the policy to maximize this reward while staying close to the original distribution (KL penalty). This dramatically reduces harmful outputs, improves instruction-following, and makes the model more conversational. The cost is significant human annotation effort, but the quality improvement is the key differentiator for ChatGPT vs raw base models.

6. Data Pipeline & Storage

Data

6.1 Data Stores

Store Purpose Why This Choice
PostgreSQL (sharded) User accounts, conversation metadata, billing ACID compliance for critical data; rich querying; mature ecosystem; Citus extension for horizontal scaling
Redis Cluster Session cache, rate-limit counters, conversation context cache, pub/sub Sub-millisecond latency; supports data structures (sorted sets for rate limiting); Cluster mode for horizontal scaling
Apache Kafka Event streaming: user interactions, model outputs, analytics Durable, high-throughput event log; decouples producers/consumers; enables replay for training data curation
S3 / MinIO Model weights, training datasets, logs, backups Cost-effective at petabyte scale; S3 API is universal; MinIO for on-prem air-gapped deployments
Elasticsearch Full-text search across conversations Inverted index for fast search; users can search their chat history semantically
🔑 Key Decision

Why Redis for Conversation Context Caching?

A conversation's recent context (last ~20 message pairs) is needed for every inference request. Fetching this from PostgreSQL on every turn would add 10-50ms of latency. By caching the serialized context window in Redis with a TTL of 1 hour (extended on each interaction), the Chat Orchestrator retrieves it in <1ms. Redis Cluster with consistent hashing ensures the cache scales horizontally. On cache miss, it falls back to PostgreSQL and repopulates Redis.

6.2 Event-Driven Architecture

All significant events (message sent, response completed, user feedback, moderation flag) are published to Kafka topics. This enables:

  • Analytics – Real-time dashboards for usage patterns, latency, error rates
  • Training Data Curation – High-quality conversations (with user consent) flow to the data warehouse for future fine-tuning
  • Billing – Token usage aggregation for metered billing
  • Audit Log – Immutable record for compliance and debugging

7. Scaling & Load Management

Infrastructure

7.1 Multi-Region Deployment

ChatGPT is deployed across multiple geographic regions (US East, US West, Europe, Asia-Pacific) to minimize latency for global users. Each region is a self-contained stack with its own inference cluster, API Gateway, and data stores. GeoDNS (Route 53 latency-based routing) directs users to the nearest healthy region.

7.2 Autoscaling Strategy

Layer Scaling Metric Mechanism
API Gateway Requests per second, CPU Horizontal Pod Autoscaler (HPA) in K8s; target 70% CPU
Chat Orchestrator gRPC request latency (p99), active connections KEDA (Kubernetes Event-Driven Autoscaling) with custom Prometheus metrics
GPU Inference Pods Queue depth, GPU utilization KubeRay autoscaler; pre-warmed GPU pods in a "warm pool"; scale-up triggered when queue exceeds threshold for >30s
🔑 Key Decision

Why a "Warm Pool" of GPU Pods?

GPU pod startup is slow (2-5 minutes for model loading into GPU memory). Traffic can spike 10× in minutes (e.g., viral social media mention). Keeping a buffer of pre-loaded but idle GPU pods (warm pool) allows instant scale-up. The trade-off is higher base infrastructure cost, but this is preferable to users experiencing multi-minute delays. The warm pool size is dynamically adjusted based on time-of-day traffic predictions.

7.3 Rate Limiting & Fairness

A token bucket algorithm implemented in the API Gateway (using Redis for distributed counters) enforces per-user rate limits. Paid tiers get larger buckets. During overload, a priority queue in the Inference Gateway ensures premium users experience lower latency while free users may see queuing. 

            // Redis Lua script for atomic rate limiting
            local key = KEYS[1]           -- rate_limit:user:{id}:minute
            local limit = tonumber(ARGV[1]) -- max requests per window
            local window = tonumber(ARGV[2])-- window in seconds
            local current = redis.call('INCR', key)
            if current == 1 then
            redis.call('EXPIRE', key, window)
            end
            if current > limit then
            return 0  -- rate limited
            end
            return 1      -- allowed

8. Security & Safety Architecture

Security

8.1 Multi-Layered Defense

Layer Mechanism Purpose
Network Cloudflare WAF, DDoS protection, IP reputation filtering Block volumetric and application-layer attacks before reaching origin
Authentication OAuth2/OIDC (Google, Microsoft, Apple) + WebAuthn for passkeys Phishing-resistant auth; no password storage liability for social logins
Authorization JWT with short expiry (15min) + refresh tokens; RBAC for internal services Stateless auth for horizontal scaling; fine-grained service-to-service permissions
Input Safety Pre-moderation classifier; prompt injection detection; PII scrubber Block harmful prompts before they reach the model; detect jailbreak attempts
Output Safety Post-moderation classifier; toxicity scoring; refusal mechanism Prevent model from generating harmful content; hallucination guardrails
Data Encryption at rest (AES-256) and in transit (TLS 1.3); field-level encryption for PII Defense in depth for sensitive user data
🔑 Key Decision

Why Separate Pre- and Post-Moderation?

Pre-moderation catches obviously malicious inputs (e.g., "how to build a bomb") before they consume expensive GPU inference time. It uses a lightweight classifier (~50ms). Post-moderation catches model outputs that are subtly harmful, hallucinated, or contain PII leakage—things only detectable after full generation. Running both provides defense-in-depth. If post-moderation flags a response, it's either replaced with a refusal message or the conversation is terminated, depending on severity.

8.2 Data Privacy

  • Conversation isolation – Strict tenant isolation at the database level; no cross-user data leakage
  • Opt-out controls – Users can disable conversation history; data not used for training
  • Data residency – Regional deployments ensure data stays within legal jurisdictions (GDPR, etc.)
  • Audit logging – All data access is logged; immutable audit trail in Kafka

9. Observability & Monitoring

Infrastructure

The system follows the three pillars of observability:

9.1 Metrics

  • Prometheus scrapes metrics from all services (request rate, error rate, latency percentiles, GPU utilization, token throughput)
  • Grafana dashboards visualize real-time system health; SLO tracking (99.9% availability, p99 latency < 2s for full response)
  • Alertmanager triggers PagerDuty alerts on SLO breaches

9.2 Logging

  • Structured logging (JSON format) from all services, shipped to Loki via Promtail
  • Correlation IDs propagate through the entire request chain (API Gateway → Orchestrator → Inference → back)
  • Log retention: 30 days hot (Loki), 1 year cold (S3)

9.3 Tracing

  • OpenTelemetry instruments all services for distributed tracing
  • Jaeger visualizes end-to-end trace for each request, showing time spent in each service, database query, and GPU inference step
  • Critical for debugging latency spikes and identifying bottlenecks

🔑 Key Decision

Why Canary Deployments for Model Updates?

New model versions can introduce regressions in quality that aren't caught by automated tests (e.g., subtly worse reasoning, new hallucination patterns). By routing 1% of traffic to the new model version and comparing key metrics (user thumbs-up/down ratio, response length distribution, toxicity classifier trigger rate, conversation completion rate) against the stable version, we can detect quality degradations before they affect all users. If metrics diverge negatively beyond a threshold, the canary is automatically rolled back. This is essentially A/B testing for model quality.

11. Key Takeaways & Design Philosophy

🎯 Core Design Principles:
  • ✅ Latency is the product – Every millisecond matters; streaming, caching, and edge deployment are not optional
  • ✅ GPU efficiency = cost efficiency – PagedAttention, continuous batching, and quantization directly impact the bottom line
  • ✅ Safety is layered, not bolted on – Pre-moderation, post-moderation, RLHF alignment, and human oversight work together
  • ✅ Observability enables velocity – Without deep tracing and metrics, debugging distributed LLM systems is impossible
  • ✅ Graceful degradation > perfect availability – Rate limiting, priority queues, and warm pools ensure the system degrades predictably under load
  • ✅ Data flywheel – Every interaction (with consent) feeds back into improving the model via RLHF and fine-tuning

This system design represents a production-grade architecture for serving LLMs at scale.
Each decision balances performance, cost, safety, and user experience — the four pillars of AI infrastructure.
