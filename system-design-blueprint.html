01
Clarify Functional Requirements
⏱ 0–5 minutes NEVER SKIP THIS Think out loud
Golden Rule: The interviewer is testing your judgment as a product engineer, not just a systems builder. Asking the right questions signals seniority. Never jump to design before you own the requirements.
"Before I design anything, let me make sure I understand what we're actually building — I'd rather spend 5 minutes on requirements than 40 minutes designing the wrong system."
Opening Script

"Great, I'd like to start by clarifying the requirements before jumping into any design decisions. I'll think out loud — please feel free to stop me or redirect. First question — what's the core user action this system enables? And are we solving a problem that's already been solved partially, or is this greenfield?"

The 5 Dimensions of Functional Clarity
Universal Clarifying Questions (Pick 3–4)
🎯 Scope tightening questions▼
"Should I design this for one geographic region or globally?" — Multi-region changes replication, CDN, compliance, and latency dramatically.

"Are we designing the full system or a specific component? For example, if this is a social network, should I design feed generation or the whole platform?"

"What should I explicitly NOT design? For example, auth, billing, analytics?"
📦 Data lifecycle questions▼
"Do users ever delete their data? What's the retention policy?" — Soft vs hard delete matters for DB design.

"Is historical data needed or only current state?" — Event log vs point-in-time snapshot architecture.

"Can the same resource be modified by multiple parties simultaneously?" — Collaborative editing is a different beast.
🔁 Feedback / confirmation framing▼
End requirements gathering with a crisp summary:

"So to confirm — the core flows are [X, Y, Z]. We're treating [auth/payment/etc.] as external black boxes. The data is primarily [type] and the system is [read/write]-heavy. Does that sound right before I move on to non-functional requirements?"
Common Mistake Anti-Patterns
⛔ Don't Do
Jump to "I'll use Kafka and Redis" before requirements
Design every possible edge case in phase 1
Ask 15 questions — pick the most impactful 5
Argue about requirements
✅ Do This
Narrow scope intentionally and say why
Write a 1-sentence system goal on your whiteboard
Keep a "out of scope" list visible
Confirm understanding before proceeding

Non-Functional Requirements
⏱ 5–10 minutes Drives all architecture Select your NFRs below
Why NFRs matter most: Two systems with identical functional requirements can have completely opposite architectures depending on their NFRs. A system that must be CP (consistent) looks nothing like one that must be AP (available). Select the NFRs that apply — the tool arsenal on the right will update.

The CAP Theorem Decision
CP — Consistency + Partition Tolerance
Banking, inventory, bookings — can't show stale balance. Sacrifice: availability during partition. Examples: HBase, Zookeeper, etcd, Postgres with sync replication
AP — Availability + Partition Tolerance
Social feeds, search, recommendations — stale is OK. Sacrifice: strong consistency. Examples: Cassandra, DynamoDB, Couchbase
CP systems refuse writes during partition to prevent inconsistency. Users may see errors. After partition heals, data is consistent everywhere. Use when data correctness is life-or-death (money, inventory counts).
SLA Targets — State These Explicitly
📊
Availability
99.9% = 8.7hr/year downtime
99.99% = 52min/year
99.999% = 5.2min/year
Each 9 is 10x harder
⏱
Latency SLA
Always state P99 not avg.
P50 hides tail latency.
P99=500ms means 1 in 100 requests takes 500ms — that's millions of users.
💿
Durability
11 nines (Amazon S3) = ~1 object lost per 100 billion per year. State your acceptable data loss window (RPO).
🔄
Recovery Time (RTO)
How long can system be down before business impact is severe? This drives redundancy architecture.
Script: Stating NFRs to Interviewer

"For non-functional requirements, I'll make a few explicit decisions that will drive the design. First, availability: I'll target 99.99% — this is a consumer-facing product. Second, latency: P99 under 200ms for read operations. Third, consistency: For [this feature] I'll choose eventual consistency — users can see slightly stale data and that's acceptable. For [financial operations] I need strong consistency. Are these assumptions reasonable, or should I adjust?"

03
Capacity Estimation
⏱ 10–15 minutes Napkin math — show reasoning Drives infra decisions
Interviewers don't expect exact numbers. They want to see that you can reason from first principles, identify the dominant constraint (is this CPU-bound? Network-bound? Storage-bound?), and arrive at a ballpark that justifies your architectural choices.
Step 1 — Traffic Estimation
Daily → Per-Second Conversion
QPS = Daily_Requests / 86,400 (≈ 100,000)
💡 Trick: 1M req/day ≈ 12 req/sec. 100M req/day ≈ 1,200 req/sec. 1B req/day ≈ 12,000 req/sec.
Peak Traffic (always account for bursts)
Peak QPS = Average QPS × 3 (conservative) to × 10 (viral/event)
💡 A Black Friday sale or viral tweet can spike traffic 10–50x. Morning rush for a ride-sharing app is 5x average. State your assumption explicitly.
Read/Write Split
Write QPS = Total × write_ratio
Read QPS = Total × read_ratio
Social: 100:1 read/write
Logging: 99:1 write/read
Messaging: ~1:1
Bandwidth
Bandwidth = QPS × avg_payload_size
1000 req/sec × 10KB = 10 MB/sec = 80 Mbps — fine. 1000 req/sec × 1MB (video) = 1 GB/sec — needs CDN.
Step 2 — Storage Estimation
Total Storage Over N Years
Storage = writes_per_day × record_size × 365 × years × replication_factor
Example: 1M new posts/day × 1KB/post × 365 × 5 years × 3 replicas = 5.5 TB. Manageable on a few nodes. Add media: 1M × 500KB × 3 = 2.7 PB — needs distributed object store.
Memory (Cache Sizing)
Cache = hot_data × avg_size
20% of data handles 80% of traffic
If 10M daily active users, ~2M "hot" user profiles. At 1KB each = 2GB of cache. Fits in RAM easily.
Quick Reference Sizes
UUID: 16 bytes
Timestamp: 8 bytes
Tweet: ~300 bytes
User profile: ~1KB
Photo thumbnail: ~50KB
HD photo: ~3MB
1-min video (HD): ~150MB
Step 3 — Derive Architecture Implications
Script: Walking Through Estimation

"Let me do a quick back-of-napkin estimate to validate my design choices. Assuming 100 million DAUs, each making 10 requests per day — that's 1 billion requests daily, roughly 12,000 QPS average. Peak might be 3–4x that — so I need to design for ~50,000 QPS. If this is 80:20 read-to-write, that's 40K reads and 10K writes per second. I'll need a caching layer for reads and likely write sharding. Storage — if each action is 500 bytes and we retain 3 years: about 500 bytes × 1B requests/day × 365 × 3 ≈ ~550 TB. That means distributed storage with replication. Does this scale roughly match your expectations?"

Define Core APIs
⏱ 15–18 minutes Short but critical
Why define APIs before drawing boxes? APIs are the contract between your system and the world. They force you to concretize exactly what data flows in and out, which prevents "fuzzy box" designs where you don't know what a service actually does.

Resource-oriented. Stateless. HTTP verbs. Best for external-facing, CRUD-heavy systems. Easy to cache GET responses.

API Definition Format (write on whiteboard)
REST Endpoint Template
POST /v1/[resource]
Auth: Bearer {token}
Body: { field1, field2, ... }
Returns: 201 { id, created_at, ... }
Always include: versioning (/v1), auth mechanism, request body shape, response shape, and error codes (400, 401, 403, 429, 500).
Rate Limiting Headers (always mention these)
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 450
X-RateLimit-Reset: 1672531200
Mentioning rate limiting in API design is a strong signal — it shows you're thinking about abuse prevention and fair usage from the very start.
Script: Defining APIs

"I'll define the 3–4 core APIs that the system exposes. I won't enumerate every field — just the meaningful ones. First, the creation endpoint: POST /v1/[resource] — takes [fields], returns an ID and timestamp. Second, the retrieval endpoint — this one I'll design with pagination in mind from day one: GET /v1/[resource]?cursor=X&limit=20, returning results plus a next_cursor. Third, the action endpoint: PUT /v1/[resource]/{id}/[action]. I'm using cursor-based pagination over offset-based because at scale, OFFSET N scans N rows before returning — terrible for deep pages. Does this API contract look right before I design the internals?"

Deep Dive + Full Tool Arsenal
⏱ 28–50 minutes (22 min) This is where you WIN or LOSE Click any tool for deep detail
The interviewer steers this phase. They'll say "tell me more about X" — follow their lead. But also proactively volunteer deep dives on the most complex/interesting parts. Never wait to be asked. Show initiative.
The Complete Tool Arsenal — Hierarchical (Click to expand + inspect)
🗄️ Database Scaling ▼
Vertical Scaling (Scale-Up)▶
Bigger CPU, More RAM, Faster SSD — first move, requires downtime, hard ceiling
Connection pooling (PgBouncer) — multiply effective connections 10x without new nodes
Read Replicas▶
Async replication → serve all reads from replicas, writes only to primary
Replication lag (50ms–5s) — reads may be stale. Acceptable for social feeds, not financial balance
CQRS pattern — separate Command (write) and Query (read) models entirely
Sharding (Horizontal Partitioning)▶
Hash-based sharding — hash(user_id) % N → uniform distribution, no hotspots, no range scans
Range-based sharding — shard by time or alphabetical — enables range scans, prone to hotspots
Directory-based sharding — lookup table maps key → shard. Flexible but single point of failure on lookup table
Consistent Hashing (with virtual nodes) — adds/removes nodes remaps only 1/N keys
Database Selection▶
PostgreSQL — ACID, joins, rich queries. Best for relational data with complex queries
Cassandra — AP, wide-column, linear write scale. Best for time-series, high-write IoT/logs
DynamoDB — fully managed, single-digit ms, auto-scale. Best for key-value at massive scale
Graph DB (Neo4j) — relationships are first-class. Best for social graph, fraud detection, recommendations
ClickHouse / Redshift — columnar OLAP. Best for analytics, aggregation over billions of rows
Elasticsearch — inverted index, full-text search, faceted search. Don't use as primary DB
⚡ Caching Strategies ▼
Where to Cache▶
CDN Edge Cache — static assets, public responses. Zero latency for user, zero load on origin
Application-level in-process cache (Caffeine, LRU Map) — no network hop, tiny subset, 5-second TTL
Distributed cache (Redis Cluster) — shared across all app servers, 100K+ ops/sec, rich data structures
Cache Write Policies▶
Cache-Aside (Lazy Loading) — read: check cache, miss → DB + populate cache. Most common pattern
Write-Through — write to cache AND DB synchronously. Always fresh, but higher write latency
Write-Back (Write-Behind) — write to cache only, async persist to DB. Fast writes, risk of data loss
Read-Through — cache sits in front of DB, transparently fetches on miss. Cache manages population
Cache Problems & Solutions▶
Cache Stampede — key expires, 10K requests all hit DB. Fix: mutex lock, probabilistic refresh, stale-while-revalidate
Hot Key Problem — single key at 500K req/sec saturates one node. Fix: local cache + key replication
Cache Invalidation — hardest problem. Event-driven (publish on write) vs TTL-based expiry
Negative Caching — cache "not found" results to prevent DB hammering for non-existent keys
📨 Async Processing & Message Queues ▼
Queue Systems▶
Kafka — distributed append-only log. Replay, consumer groups, ordered within partition. At-least-once
Amazon SQS — fully managed, at-least-once, FIFO option. No replay. Simpler operationally
RabbitMQ — complex routing (exchanges, bindings), per-message ACK, dead letter queues
Delivery Guarantees▶
At-Most-Once — fire and forget. May lose messages. OK for metrics, logs, non-critical events
At-Least-Once — producer retries until acked. May duplicate. Require idempotent consumers
Exactly-Once — idempotent producer + transactional consumer. Most complex. Use for financial events
Patterns▶
Fan-out — one producer event consumed by N independent consumer groups simultaneously
SAGA Pattern — distributed transaction as a series of events with compensating rollback transactions
Transactional Outbox — write event to DB table in same transaction, relay service forwards to queue
Dead Letter Queue — poison pill messages that repeatedly fail get routed to DLQ for manual inspection
📡 Real-Time Communication ▼
Protocols▶
WebSocket — persistent full-duplex. Best for chat, live tracking, collaborative editing. Stateful connections
SSE (Server-Sent Events) — server-to-client one-way push. Simpler than WS. Best for live feeds/notifications
Long Polling — client holds open HTTP request until event or timeout. Simplest to implement, higher overhead
WebRTC — peer-to-peer UDP. Best for video/audio. Low latency, complex NAT traversal (STUN/TURN)
Scaling Real-Time▶
Sticky sessions — WS connections must stay on same server. L4 LB with consistent hashing by connection
Pub/Sub broker (Redis Pub/Sub) — decouple WS server from event source. Fan-out to N subscribers
Presence system — track online/offline status with heartbeat + Redis TTL per connection
🔥 High Availability & Resilience ▼
Replication▶
Primary-Replica — all writes to primary, async sync to replicas. Failover via promotion
Multi-Master — writes to any node, conflict resolution required. Higher availability, complex
Multi-Region Active-Active — serve each region locally, async cross-region replication
Failure Handling▶
Circuit Breaker — stop calling a failing service. OPEN/HALF-OPEN/CLOSED states. Prevent cascading failure
Retry with exponential backoff + jitter — never retry immediately. Add jitter to prevent synchronized retry storms
Bulkhead — isolate failures in one service from cascading to others. Separate thread pools per dependency
Graceful Degradation — serve cached/stale data instead of errors. "Feed may be delayed" > 500 error
Health checks — /health endpoint. Liveness (is process alive?) vs Readiness (is it ready to serve?)
Consensus Protocols▶
Raft — leader election, log replication. Used in etcd, CockroachDB, Kafka KRaft. Understandable
Paxos — original consensus algorithm. Notoriously complex. Basis of Chubby, Spanner
Gossip Protocol — epidemic-style state propagation. Used in Cassandra, Dynamo. No central coordinator
💾 Storage Patterns & Durability ▼
Storage Types▶
Object Store (S3) — immutable blobs, infinite scale, cheap, 11-nines durability. Media, backups, archives
Block Store (EBS) — attached disk for DBs needing fast random I/O. Tied to one instance
Network File System (EFS/NFS) — shared filesystem across instances. Best for shared config or build artifacts
Time-Series DB (InfluxDB, TimescaleDB) — metrics, sensor data. Optimized for time-range queries
Durability Mechanisms▶
Write-Ahead Log (WAL) — write to append-only log before applying to data structure. Crash recovery
Checksums — detect silent data corruption at storage layer. Always stored with data
Snapshots — point-in-time copy of state. Basis for backups and fast recovery
Data Partitioning▶
Hot/Cold tiering — recent data on SSD, old data on HDD/S3. Transparent to application
Log Compaction (Kafka) — retain only latest value per key. Turns log into current-state snapshot
Columnar storage (Parquet) — store by column not row. 10x compression for analytics workloads
🔒 Consistency & Distributed Transactions ▼
Consistency Models▶
Strong Consistency — all reads see latest write. Requires synchronous replication. Slow but correct
Eventual Consistency — all nodes will converge eventually. Fast, available. Accepts temporary staleness
Read-Your-Writes — a user always sees their own writes immediately (sticky read to primary for that user)
Distributed Transactions▶
Two-Phase Commit (2PC) — coordinator + participants. Atomic across nodes but blocking on coordinator failure
SAGA — sequence of local transactions with compensating actions for rollback. No blocking. Eventual
Idempotency keys — make retries safe. API key + DB unique constraint prevent duplicate operations
Optimistic Locking — version field in row. On write, check version matches. Conflict = retry. No DB lock held
🔭 Observability & Operations ▼
The 3 Pillars▶
Metrics (Prometheus/Datadog) — latency histograms, error rates, saturation, traffic (RED method)
Structured Logging (ELK Stack) — JSON logs with trace ID, user ID, operation, duration, result
Distributed Tracing (Jaeger/Zipkin) — trace request across all microservices. Find slow components
Rate Limiting▶
Token Bucket — allows bursts up to bucket size. Tokens refill at fixed rate. Most common
Sliding Window Counter — smooth rate tracking. More accurate than fixed window. Redis ZADD implementation
Leaky Bucket — process requests at constant rate. Smooths out bursts. Queue-based
Deep Dive Trigger Phrases
"The most interesting part here is [X] — let me go deeper on that because there are some non-obvious tradeoffs..."
"I want to flag a potential bottleneck proactively — the [component] could become a hotspot because..."

"I'd normally use [X] here but given the [specific constraint], I'll instead use [Y] because..."
07
Bottlenecks & Trade-off Analysis
⏱ 50–55 minutes This separates seniors from juniors
"Every architectural decision is a trade-off. Senior engineers articulate what they sacrificed, not just what they gained."
Bottleneck Identification Framework
Trade-off Framework — State These Explicitly
⚖️
Consistency vs Availability
Every distributed storage decision. During partition: do we reject writes (CP) or accept them risking divergence (AP)?
📏
Latency vs Throughput
Batching increases throughput but hurts individual request latency. A 5ms batch window can increase throughput 100x.
💸
Cost vs Performance
In-memory storage is 100x faster but 100x more expensive than SSD. Cache only the hot 20% of data.
🔄
Simplicity vs Resilience
Every redundancy mechanism adds operational complexity. 5-nines availability requires an on-call team. Know what you're signing up for.
🕰️
Freshness vs Performance
Caching improves performance but serves stale data. State your acceptable staleness window explicitly for each data type.
🔧
Flexibility vs Efficiency
Schemaless NoSQL is flexible but queries are limited and joins are impossible. Schema-on-write enables richer queries but requires migrations.
Script: Proactive Bottleneck Review

"Let me do a quick bottleneck pass before we close. Single points of failure: the database primary is a SPOF — I'd mitigate with a hot standby and automated failover (RDS Multi-AZ). The message queue leader — Kafka handles this with ISR replication and controller re-election. Hotspots: if a celebrity uses this system, their data key will get hammered — I'd use the local cache + key replication pattern I described earlier. Biggest trade-off I'm making: I chose eventual consistency for the feed, which means a user might see a post appear out of order by a few seconds. The alternative was synchronous replication which would add 50-100ms to every write — given that we're optimizing for write throughput, I consider that trade-off acceptable. Would you like me to revisit any of these decisions?"

Wrap Up & Extensions
⏱ 55–60 minutes Leave them wanting more
Wrap-Up Checklist (Last 5 Minutes)
System Design Anti-Patterns (Never Do These)
⛔ Fatal Mistakes
Using a single global DB without mentioning replicas or sharding
Designing without mentioning failure modes at all
"Just use Kafka" without explaining why
Ignoring the N+1 query pattern
Never asking what scale means to the interviewer
Using microservices for a simple CRUD app ("over-engineering trap")
Designing sync everything — never showing async understanding
Never mentioning cache invalidation (the hardest part!)
✅ Senior Signals
Leading with trade-off rationale
Separating hot/cold paths proactively
Mentioning idempotency unprompted
Designing for operational simplicity, not just theoretical elegance
Naming specific failure modes and degradation behavior
Correct latency intuitions (Redis=μs, network=ms, disk=ms–s)
Distinguishing P50 vs P99 latency
Mentioning observability without being asked
Latency Intuition — Memorize These
Numbers Every Engineer Should Know (approximate, 2024 hardware)
L1 cache: 0.5ns | L2 cache: 7ns | RAM: 100ns SSD random read: 150μs | HDD seek: 10ms Compress 1KB: 3μs | Send 1KB over 1Gbps: 10μs Redis GET: 0.1–0.3ms | DB query (indexed): 1–10ms Same DC network RTT: 0.5ms | US→EU: 75ms | US→Asia: 150ms
💡 Key rule: Memory is 1000x faster than SSD. Network same-DC is 5x faster than SSD. Cross-continent is 1000x slower than same-DC.
